# PROPOSAL: 액션 분류 — "내 응답 대기"·"기한"을 쓸 수 있는 수준으로

> 상태: **구현 완료(2026-07-17)**. 목적(사용자 확정): 의미가 없던 자동 분류
> ("↩ 내 응답 대기"·"⏰ 기한/요청")를 AI 없이, 오탐을 줄이되 **미탐을 조용히
> 늘리지 않으면서** 실사용 가능한 수준으로.

## 0. 문제 (2026-07-17 분석에서 확인)

| # | 증상 | 원인 |
|---|---|---|
| 1 | 홈과 웹 ↩ 탭의 '응답 대기'가 서로 다름 | 웹은 본문 무검사(마지막 수신 + To<3 + 나), 홈은 별도 규칙 |
| 2 | "잘/참고/양해 부탁드립니다"가 요청으로 오탐 | `부탁드립` 앵커 없는 매치 |
| 3 | "검토 요청 건을 완료했습니다"가 고신뢰 '결정 대기' | 완료·과거 문맥 무시 |
| 4 | ⏰ 가 회신 후에도 영구히 남음 | `deadline_count` 누적만, 감소 경로 없음 |
| 5 | "notification …" 실무 제목이 강한 노이즈로 사망 | 제목 부분 문자열 매치 |
| 6 | 영문 요청·전각 ？ 미탐 | 어휘 부재 |

## 1. 구조 — 판정 3층 (본문 재스캔 없음)

```
L1 message_features   수집 시 1회, 문장 단위 게이팅 (features.classify_message)
L2 thread_state       열린 요청 슬롯 상태기계, 수집 트랜잭션 증분 (store.fold_action)
L3 actions.py         열린 슬롯 × 수신 대상 × 노이즈 → REQUIRED/MAYBE/NONE
```

- **L1 문장 게이팅**: 요청·기한은 같은 문장에 완료/과거 문맥이 있으면 무효.
  순서 무관(문장별 판정 후 OR) — "마지막 의도 채택" 방식은 한국어 맺음말
  ("잘 부탁드립니다"가 항상 마지막)에 깨져서 채택하지 않았다. '다시/재차/
  리마인드'는 과거 언급이라도 재요청으로 살린다. 강한 요청(행동 동사 앵커)과
  약한 요청(맨 부탁/확인 바랍니다)을 분리 저장. 인사 관용구(참고/양해/잘 부탁)는
  약한 요청도 아니다.
- **L2 상태기계**: 스레드당 열린 액션 슬롯 1개(의도적 단순화 — 최신 요청이
  우선한다는 스레드 단위 도구의 기존 철학). 전이는 `fold_action` 순수 함수
  하나로, 증분(_update_thread_state)·역순 삽입 refold·백필이 전부 같은 함수를
  써서 **증분 ≡ 전체 재계산**이 정의상 성립(무작위 순서 삽입 drift 테스트 가드).
  - 해소: **내 실질 발신**(++수신인 추가·빈 본문 등 trivial 제외)과 **명시적
    철회**("회신 불필요")만. **상대의 완료 통보는 해소가 아니다** —
    `completion_after_action` 표시 후 MAYBE 강등까지만. 오탐(불필요한 알림)과
    미탐(놓친 공)의 비용이 비대칭이라, 잘못 닫힘은 조용한 실패가 되기 때문.
  - 재촉이 약해도 격하 없음: 열린 창 안에서 강도·decide·기한은 단조, source 만
    최신 요청으로 갱신.
- **L3 사다리** (위가 먼저):
  1. 숨김·확실한 노이즈(차단/자동발송 발신, 강한 제목) → NONE
  2. 강한 요청 × 직접 수신/이름 지목/내 참여 → REQUIRED;
     그룹 To·CC(지목 없음)/리스트 수신 → MAYBE.
     약한 신호는 질문+직접, 제목 요청 표지+직접, 이름 지목+요청만 REQUIRED,
     나머지 직접 수신 → MAYBE, 그 외 → NONE
  3. 강등(전부 **MAYBE 까지만** — NONE 으로 안 떨어뜨림): 요청 후 완료 통보 /
     대량 발송(지목 없음) / 외부 발신(허용 목록 외).
     수신인 수는 하드 제외가 아니라 여기의 강등 요소로만.
  - 모든 판정은 reason code 를 갖고(설명 가능성), 근거 문장은 표시 시 원본
    메시지에 정규식을 재실행해 복원한다(추가 저장 없음 — 비트=게이트 원칙).

## 2. 화면 통합 — 불일치 제거

- 홈 개입 큐(decide/respond) = 웹 ↩ 탭 = 스레드 상세 신호 = **같은
  `actions.classify_threads`** — 일치율 100%가 구현이 아니라 정의.
- **MAYBE = 접힌 '확인 후보'** (홈 + ↩ 탭 하단). 부정 규칙의 목적지를 제거가
  아니라 여기로 — 잘못 강등된 공을 사용자가 훑어서 회수한다. AI(haiku) 분류
  후보 풀도 이것으로 대체(같은 경계 항목).
- **⏰ = 열린 액션의 기한** — 내 회신·철회로 액션이 닫히면 함께 사라진다(#4
  해결). DEADLINE_RX 의 요청 프록시('회신 부탁'·'주시면')는 요청 계층으로
  분리해 ⏰ 가 ↩ 의 복제가 되지 않게 했다(탭 이름도 '기한/요청'→'기한').
- 상세 화면: `판정: 직접 수신 + 명시 요청 — 「내일까지 검토 부탁드립니다.」`
  처럼 이유+근거 문장 표시.

## 3. 노이즈 정책

- **hard**(즉시 NONE): 차단·자동발송 발신, 강한 제목. 강한 제목은 부분 매치
  폐지 → `[태그]` 포함 / `패턴:`·`[패턴]` 형태 / 답장 접두 제거 후 전체 일치만
  ("notification 설정 변경 검토 요청" 실무 제목 생존, #5).
- **policy**(강등만): `internal_domains` 켠 상태의 외부 발신. 새
  `filters.external_allowlist`(협력사 도메인/주소)로 전부-아니면-전무 구조 해소.

## 4. 배포·무재싱크

`FEATURE_VERSION=2` + `_derived_version` 에 **my_names 해시 추가**(이름이 저장
비트 `mentions_me` 의 입력이 됐으므로). 버전 불일치 시 파생 테이블을
**drop+재생성**(SQLite ADD COLUMN IF NOT EXISTS 부재 우회 — 파생 테이블은
messages 에서 재구축 가능) 후 1회 백필. Outlook 재수집·DB 삭제 불필요 —
`git pull` 후 서버 시작 시 자동.

## 5. 측정 (합성 수치는 방향 지표 — 실측은 audit 라벨로)

- 합성 코퍼스(구 측정과 동일): 기한 검출 44% → **88%**, 요청 100%, 과잡 0.
  남은 미탐 2건은 알려진 한계("늦어도 수요일에는", "회의 전까지"의 비인접).
- **독립 라벨드 코퍼스**(tests/synthetic_corpus.py, 490 스레드/1,020통 —
  작성자와 판정기 구현이 분리된 평가): REQUIRED **정밀도 1.00 · 재현율 0.97 ·
  F1 0.985**, gold-NONE 330건 오탐 0. 놓침 3건은 전부 'CC 수신 + 전원 지목'
  사례로 MAYBE(확인 후보)에 남아 회수 가능. 이 평가가 잡아준 규칙 구멍:
  "기존 요청은 무시해 주세요"(철회인데 강한 요청으로 오인), "문제 없으면 ~"
  (조건문을 완료로 오인), "가능하면 ~해 주시면"(완곡 요청 강등), 약신호+그룹
  수신의 NONE→MAYBE, 외부+약신호(뉴스레터)의 MAYBE→NONE.
- 분류 비용(1.7KB 본문): 0.32→0.57ms/통, ingest 0.65→0.96ms/통 — 수집
  트랜잭션 락 보유 시간 기준 안전(일일 수백 통 ≈ 1초 미만). 병리 입력(3만 자
  단일 줄·미끼 반복) 최대 13ms — 백트래킹 폭발 없음.
- 백필 3,000통 1.85s (≈ 1만 통 6s, 1회성).
- 증분 ≡ refold ≡ 백필 등가: 무작위 순서 80~120통 drift 0 (정식 테스트 편입).

**실측 루프**: `mailkb audit` (PC 실 DB) — 최신 스레드의 판정+이유+근거 출력,
`--label` 로 정답을 `<home>/labels.jsonl` 에 축적(개인정보는 data/ 원칙),
`--report` 가 라벨 × 현재 판정 혼동 행렬을 낸다. **이후 모든 규칙 변경은 이
라벨 집합으로 개선/후퇴를 측정한다** — 합성 수치는 회귀 가드일 뿐.

## 5-b. 리뷰 반영 (2026-07-17, 2차)

교차 리뷰가 확인한 결함 2건 수정:

1. **hard 노이즈의 액션 오염(치명)** — 자동회신("7월 20일까지 부재…부탁드립니다")
   이 열린 요청의 source 를 탈취해 L3 hard_noise → NONE 으로 만들거나, 시스템
   "완료" 문구가 completion_after_action 강등을 일으켰다(재현 후 수정). fold 의
   세 경로(증분·refold·백필) 모두 hard 노이즈 메시지를 전이 대상에서 제외하고,
   노이즈 표면(ignore/blocked/subject_strong)을 `_derived_version` 에 포함해
   차단 목록 변경 시 자동 재접기한다. Store 는 noise 판정자(Config)를 주입받는다.
2. **근거 추출의 게이트 불일치** — `sentence_gate()` 를 features 로 분리해
   classify(저장 비트)·evidence(근거 표시)·deadline_signals(데일리)가 같은
   게이트를 쓴다. "검토 요청 건을 완료했습니다"가 근거 문장으로 표시되지 않는다.
3. 외부+약신호는 MAYBE 가 아니라 NONE — 뉴스레터("오늘까지 신청…") 제외.
   협력사는 external_allowlist 가 이 분기를 안 타게 한다(강한 증거는 여전히
   MAYBE 로 남아 allowlist 등록 전에도 회수 가능).

**라벨 정의 3건 — 사용자 확정(2026-07-17)으로 종결**:
① 관용구("참고/협조 부탁드립니다" 22건) = **NONE** — 이 오탐이 작업의 발단.
  MAYBE 로 되살리면 확인 후보의 신호 대 잡음비가 무너진다. 코퍼스 라벨 수정
  (pleasantry-* 키로 분리).
② CC 수신 + 전원 지목 3건 = **MAYBE** — CC 는 참조 관례, 확인 후보로 회수
  가능. 실메일 라벨에서 놓침이 확인되면 그때 승격. 코퍼스 라벨 수정.
③ 노이즈 정의 통일 + 축 배선 — hard = 명시 설정 차단(ignore/blocked/강한
  제목)만, 외부 뉴스레터는 **policy**(차단 등록 시 hard 가 '되는' 것). 코퍼스
  10건 재라벨. 노이즈 축을 상시 회귀 게이트로 배선(tests.test_synthetic_corpus
  TestClassifierGate — 액션 축 490/490·노이즈 축 490/490 완전 일치가 기준선).
  일정 축은 기능 구현 시 배선.

## 6. 알려진 한계 (의도된 트레이드오프)

- 스레드당 액션 슬롯 1개 — 한 스레드에 미결 요청이 2개면 최신 것만 추적.
- 기한 날짜 파싱 없음 — '기한 경과(EXPIRED)'는 후속(읽기 시점 오버레이로만
  구현할 것 — 시간 경과는 ingest 이벤트가 아니라 저장 상태가 될 수 없다).
- "늦어도 X에는"·"회의 전까지"(시간어-어미 비인접) 미탐.
- 내 "감사합니다" 한 줄 회신도 해소로 침 — 구 동작과 동일(발신=이탈), 회귀 아님.
- 배포 리스트 수신(주소에 내가 없음)은 REQUIRED 불가, MAYBE 까지 — 리스트
  멤버십을 알 수 없어서.
- 일정(EVENT/CHANGE/CANCEL) 분류는 별도 트랙 — 이 배치는 오탐 감소가 목적.
