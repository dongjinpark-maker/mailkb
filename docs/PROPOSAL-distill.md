# PROPOSAL: 지식 증류 계층 (distill) — 설계안 v1.1 (방향 확정)

> 상태: **Phase 1 구현 완료 (2026-07-12)** — 스키마 + 데일리 수확 전환 + 결정 원장
> (검토 큐·수동 기록·'기록' 메뉴, 주간/분기 탭은 Phase 2 에서 추가).
> 1~2주 실데이터로 수확 품질 확인(logs/harvest.jsonl + ANALYZE-harvest.md) 후 Phase 2.
> 확정 사항: ① 데일리=수확+델타 전환 ② 원장 확정=사람 검토(휴먼 인 더 루프)
> ③ nav '기록' 개편+'프로젝트' 신설 ④ Ask 기본 백엔드=sonnet.
> 대상 기능: 결정 원장 · 인물 도시에 · 프로젝트 도시에 · 증류 피라미드 · Ask Minerva ·
> 인물 업무 브리핑 + **데일리 분석 재설계**(전제).
> 예산: 주 1회 on-demand Opus 소액 허용(사용자 확정 — 실값은 운영 노트(비공개)). 데일리는 기존 비용(sonnet/haiku) 유지.

## 0. 전제 진단 — 지금 데일리 분석이 도움 안 되는 이유

| 문제 | 원인 |
|---|---|
| (a) 매일 비슷한 말 반복 | 재료가 '스레드 롤링 요약'의 재탕 — 새 정보가 없음 |
| (b) "놓쳤을 가능성"이 공허 | 근거 없는 추측을 채우도록 프롬프트가 강요 |
| (c) 읽고 나면 끝 | 산출이 어디에도 **축적되지 않음** — 다음날이면 무가치 |
| (d) 개입 큐와 중복 | 액션은 이미 결정론 큐가 더 정확히 잡음 |

**결론: 데일리 AI의 임무를 "통찰 생산"에서 "수확(harvest)"으로 바꾼다.**
통찰(추세·리스크·교차 판단)은 컨텍스트가 넓은 **주간 Opus 증류**로 이동한다.

## 1. 큰 그림 — 2단 리듬 + 원장 3종

```
매일 (sonnet, 기존 비용)                 주 1회 on-demand (opus, $2~3)
─────────────────────────              ─────────────────────────────
델타 브리프 (어제와 달라진 것 3~6줄)      ① 프로젝트 배정·도시에 갱신
결정 후보 추출  ──────────┐             ② 인물 도시에 갱신
인물/프로젝트 신호 추출 ───┼─→ 원장 적재 ③ 결정 원장 정리(병합·확정·supersede)
개입 큐 (기존 결정론+haiku)│   (candidate)④ 주간 회고 md (업무보고 재료 포함)
                          └─────────────→ 주간이 후보를 소화·확정

원장 3종 (SQLite, 영구 축적):
  decisions        결정 원장 — 무엇을·왜·누가·언제 (+supersede 체인)
  projects         프로젝트 도시에 — 스레드 묶음의 상태·요약·타임라인
  people_dossier   인물 도시에 — 역할·진행 중·미결·관계 수치
```

**공통 원칙** (기존 철학 유지)
- 결정론이 뼈대, AI는 그 위의 서술만. AI 실패 시 이전 산출물 유지(graceful).
- **환각 가드**: 모든 추출 항목에 원문 인용(quote) 필수 → 해당 스레드 `new_content`
  부분일치 검증 실패 시 폐기. 스레드 번호 없는/모르는 항목 폐기(기존 패턴).
- **반복 방지**: 이전 산출물(어제 델타·기존 도시에)을 프롬프트에 주고 "달라진 것만" 지시.
- 증분 마커(`sync_state`·basis 카운트)로 재실행 시 신규분만 처리.

## 2. 비용 모델

가정: Opus API $5/M 입력·$25/M 출력, 한글 1자≈1토큰(보수). 구독(Max) headless면 $0.

| 실행 | 입력 | 출력 | 추정 |
|---|---|---|---|
| 주간 증류(스테이지 4~5개 합) | 스레드 요약 3~5만자 + 데일리 추출 2만 + 기존 도시에·원장 3만 + 선별 원문 5만 ≈ **15만 tok** | 도시에·회고 ≈ **3만 tok** | **$1.5~3** |
| 분기 회고(3개월 1회) | 주간 md 12장 + 원장 ≈ 8만 tok | 1.5만 tok | ~$0.8 |
| 인물 업무 브리핑(가끔) | 1인분 재료 ≈ 3~5만 tok | 1만 tok | ~$0.5 |
| 데일리 | 기존과 동일 (sonnet 1콜 + haiku 2콜, 프롬프트만 교체) | | 불변 |

실행 전 입력 자수로 예상 비용을 화면에 표시하고, 초과 시 스레드 상한 자동 축소.

## 3. 기능별 설계

### 3-A. 데일리 재설계 (전제 — Phase 1)

새 산출 스키마 (마크다운 고정 형식, 파서가 원장에 적재):

```
# 2026-07-12 데일리
## 오늘 델타          ← 어제와 달라진 것만 3~6줄 (사람이 읽는 유일한 산문)
## 결정 후보          ← 항목: 결정 한 줄 | 근거 | 결정자 | #스레드 | "원문 인용"
## 인물 신호          ← 역할·담당 변화: 이름 | 신호 | #스레드 | "인용"
## 프로젝트 신호      ← 상태 변화: 스레드 | 이전→이후 | "인용"
## 개입 필요          ← 기존 결정론 큐 (불변)
## 오늘 메일 핵심     ← 기존 디제스트 (불변)
```

- **삭제**: "놓쳤을 가능성"·"Side Effect 경고" — 주간 회고의 '리스크' 섹션으로 이동
  (원장·프로젝트 상태를 근거로 판단하므로 지금의 추측보다 정확).
- 결정 후보 → `decisions(status='candidate')`, 인물/프로젝트 신호 → 각 원장의
  pending 델타 테이블에 적재(주간이 소화). quote 검증 실패 항목은 버림.
- 홈 렌즈 '결정'은 md 파싱 대신 **원장을 직접 조회** (candidate+confirmed).
- 어제 데일리의 '오늘 델타'를 프롬프트에 넣어 "이미 보고한 것 반복 금지".

### 3-B. 결정 원장 (#1 — Phase 1)

```sql
CREATE TABLE decisions (
  id INTEGER PRIMARY KEY,
  thread_id INTEGER NOT NULL,
  decided_on TEXT,            -- 결정 시점(메일 날짜)
  title TEXT,                 -- 결정 한 줄
  rationale TEXT,             -- 근거
  decider TEXT,               -- 결정자 (이름 또는 주소)
  quote TEXT,                 -- 원문 인용 (환각 앵커 — new_content 부분일치 필수)
  status TEXT DEFAULT 'candidate',  -- candidate|confirmed|superseded|rejected
  superseded_by INTEGER,      -- 이 결정을 뒤집은 후속 결정 id
  source TEXT DEFAULT 'daily',-- daily|weekly|manual
  created TEXT
);
```

- 적재 경로 3개: ① 데일리 후보 ② 스레드 상세의 **"결정으로 기록" 버튼**(수동 —
  제목·인용 자동 채움 폼, 즉시 confirmed) ③ 주간 정리 제안.
- **확정은 사람** (2026-07-12 확정 — 휴먼 인 더 루프):
  - '기록 > 결정' 탭 상단에 **검토 큐** — "검토 대기 N건" 배지, 항목마다
    [확정] [반려] [수정 후 확정] 버튼. 한 건 처리 = POST 1회 (기존 액션 패턴).
  - 주간 Opus 는 **제안만**: 후보 간 중복 병합 제안 · 결정 아닌 것 반려 제안 ·
    supersede 체인 제안을 후보에 주석으로 붙인다(status 는 candidate 유지).
    사람은 주석을 보고 원클릭 처리 — "정확성은 사람, 노동은 AI" 분담.
- 웹: '기록' 메뉴(아래 3-E)의 **결정 탭** — 검토 큐(상단) + 확정 원장(시간 역순,
  상태 필터, LIKE 검색, #스레드 링크, "→ 이 결정으로 대체됨" 체인 표시).
- 홈 렌즈 '결정'은 confirmed 를 기본으로, 검토 대기 수를 배지로 병기.

### 3-C. 프로젝트 도시에 (#10 — Phase 2)

```sql
CREATE TABLE projects (
  id INTEGER PRIMARY KEY, name TEXT, status TEXT DEFAULT 'active', -- active|dormant|done
  summary_md TEXT DEFAULT '', first_date TEXT, last_date TEXT, updated TEXT);
CREATE TABLE thread_projects (
  thread_id INTEGER PRIMARY KEY, project_id INTEGER,
  source TEXT DEFAULT 'ai');  -- ai|manual — manual 은 AI 가 절대 안 덮음
```

- **배정(주간 스테이지①)**: 입력 = 최근 활동 스레드 목록(제목·참여자 상위 3·요약 1줄)
  + 기존 프로젝트 명단 → 출력 = `#스레드→프로젝트` 배정표 + 신규 프로젝트 제안.
  순수 결정론 클러스터링(자카드)은 오분류가 잦아 AI 배정 + 수동 교정으로 간다.
- **도시에 갱신(스테이지②)**: 프로젝트별 소속 스레드 요약 + 결정 + 미결 → 기존
  summary_md 를 주고 델타만 반영해 재작성.
- 웹: nav 에 **프로젝트** 메뉴 — 좌: 프로젝트 목록(활동순·상태 배지), 우: 도시에
  (요약 · 최근 움직임 · 관계자 · 결정 목록 · 미결 · 스레드 목록).
  스레드 상세에 프로젝트 배지 + 변경 드롭다운(→ source='manual').

### 3-D. 인물 도시에 (#3 — Phase 2)

```sql
CREATE TABLE people_dossier (
  addr TEXT PRIMARY KEY, dossier_md TEXT, updated TEXT,
  basis_msg_count INTEGER DEFAULT 0);  -- 증분 가드 (요약과 동일 패턴)
```

- 주간 스테이지③: 이번 주 교신 상위 ~15명 중 basis 이후 메시지가 늘어난 사람만.
  재료 = 참여 스레드 요약 + 결정(결정자=그 사람) + **결정론 수치**(교신량·응답
  중앙값·마지막 접촉 — report 계산 재사용).
- person 페이지 상단 카드: 역할·조직 추정 / 지금 함께 진행 중인 것 / 서로의 미결 /
  수치. 도시에 없으면 카드 자체를 안 그림.

### 3-E. 증류 피라미드 (#8 — Phase 2·3)

- **주간 회고**(주간 스테이지⑤): `vault/weekly/2026-W28.md`
  섹션: 이번 주 결정(원장 링크) / 프로젝트 움직임(도시에 델타) / **내가 한 일**
  (발신 + 결정자=나 + 종결 스레드 기반 — 업무보고 재료) / 다음 주 미결 / 리스크
  (구 Side Effect 의 자리 — 원장·프로젝트 근거로 판단).
- **분기 회고**(Phase 3, on-demand): 주간 md 12장 + 원장 → `vault/quarterly/2026-Q3.md`.
- 웹: nav '데일리' → **'기록'** 으로 개편 — 탭: 데일리 | 주간 | 분기 | 결정.

### 3-F. Ask Minerva (#6 — Phase 3)

- 검색 페이지에 모드 추가: [검색] [AI에게 묻기].
- 파이프라인: 질문 → (결정론) 조사 제거·토큰 분해 → FTS(trigram) OR 검색 상위 12
  스레드 + 원장·도시에 LIKE 매치 → **컨텍스트 팩 ≤2.5만자** (우선순위: 구조화 지식
  → 스레드 요약 → 상위 3개 스레드 원문 발췌) → sonnet 1콜(질문창에 "Opus 로"
  체크 옵션) → 답변 + 근거 #스레드 링크 목록. "재료에 없으면 모른다" 고정 지시.
- FTS 빈약 시 폴백: haiku 1콜로 검색어 3~5개 생성 후 재검색.
- 실행은 리뷰 생성과 같은 백그라운드 잡 + 상태 폴링 패턴 재사용. 우측 패널에 표시.

### 3-G. 인물 업무 브리핑 (#9 — Phase 3, 원장·도시에 의존)

**목적 (2026-07-12 재정의)**: 업무를 넘기는 인수인계가 아니라, **내가 매니저/
관찰자로서 협업자 한 명의 업무를 파악**하는 것. "이 사람 지금 뭘 하고 있고,
어디서 막혀 있나"를 메일 근거로 재구성한 심층 문서.

- 인물 도시에(3-D)와의 관계: 도시에 = 주간 자동 갱신되는 **짧은 카드**,
  브리핑 = person 페이지 버튼/CLI 로 **on-demand 생성하는 심층 문서**(Opus).
- 진입점: person 페이지 "업무 브리핑 생성 (Opus)" 버튼 + `mailkb brief <이름|주소>`.
- 재료(결정론 조립): 그 사람이 참여한 스레드의 롤링 요약 + 소속 프로젝트 도시에 +
  결정 원장(결정자=그 사람 / 그 사람 관련) + 미결(그 사람↔나, 그 사람↔관찰 가능한
  제3자) + 통계 수치(교신량 추이·응답 패턴·조용해짐 신호 — report 계산 재사용)
  + 최근 원문 선별 발췌.
- 산출 구조 (`vault/notes/brief-<이름>-<날짜>.md`):
  1. 한눈 요약 — 역할·담당 영역 추정
  2. 진행 중인 사안별 상태 — 사안: 무엇을 · 어디까지 왔나 · 다음 단계 · 막힌 곳
  3. 관여한 결정 (원장 링크)
  4. 대기·병목 — 그 사람이 기다리는 것 / 그 사람에게 걸려 있는 것
  5. 협업 수치 — 교신 추이·응답 패턴 (결정론)
  6. **관찰 한계 명시** — "내 메일함에 보이는 범위" 기준임을 문서에 고정 문구로.
     CC 로만 걸린 스레드는 불완전하다는 캐비앳 포함.
- 톤 지시(프롬프트 고정): 평가·인사판단 어휘 금지, 업무 사실 서술만. 추측은
  "~로 보임(근거 #스레드)" 형태로만.
- 비용: 1인분 재료 ≈ 3~5만 tok → Opus 1콜 ~$0.5. `--no-ai` 면 결정론 조립만.
- (파생) 프로젝트 단위 인수인계 문서는 같은 조립기의 `--project` 변형으로 추후 가능.

## 4. 주간 증류 잡 (오케스트레이션)

- 트리거: 웹 홈 **"주간 증류 (Opus)"** 버튼 + CLI `mailkb distill`. (on-demand —
  스케줄러 등록 안 함. 사용자가 주 1회 누른다.)
- 실행 전 입력 규모 → 예상 비용 표시. 스테이지: ①프로젝트 배정 ②프로젝트 도시에
  ③인물 도시에 ④원장 정리 **제안**(확정은 사람 — 3-B) ⑤주간 회고 — 각각 별도 콜,
  기존 진행표시(_StageProgress·웹 status) 재사용. 마커 `last_distill` — 실패한
  스테이지부터 재시도. 주간 회고의 '이번 주 결정'은 confirmed 만 인용(후보는 수 표기).
- 백엔드: `config._BUILTIN_BACKENDS["opus"] = ["claude","-p","--model","opus"]`,
  라우팅 키 `[ai] weekly = "opus"` (cfg.opt 로 읽음 — config 무수정 동작).

## 6. 구현 순서 (각 Phase 끝마다 실사용 검증 후 다음)

- **Phase 1** — 스키마 + 데일리 재설계 + 결정 원장(후보 적재·수동 기록·기록 메뉴).
  Opus 불필요, 비용 불변, 즉시 가치. **여기서 1~2주 실데이터로 후보 품질 확인.**
- **Phase 2** — distill.py 주간 잡(스테이지 5종) + 프로젝트·인물 도시에 + 주간 회고.
  겸사 정리: **홈 'AI 정리' 입력창·버튼 제거**(/refine·구 /lens/intervene/refine 라우트
  포함) — 유일한 고유 가치가 '메모 반영 재실행'인데 클릭 수단(추적 제외·숨김·플래그)과
  ai-rules.md 가 대체함(2026-07-12 사용자 확정). **자동 주석(run_ai_layer ⑤,
  intervention_ai 저장→홈 표시)은 유지.**
  겸사 결정: **'오늘 리뷰 생성'(비-AI) 버튼 정리 여부** — 수확 전환 후 내용이 홈에서
  이미 보이는 것의 스냅샷에 가까워짐. Phase 1 실사용에서 누른 적 없으면 'AI 리뷰
  생성' 하나로 축소(결정론 폴백은 AI 실패 시 자동 동작이라 유지됨). CLI `review`
  (비-AI)는 스케줄러·검증용이므로 유지.
- **Phase 3** — Ask Minerva + 분기 회고 + 인물 업무 브리핑.

## 6-1. 구 검토안 승계 (PROPOSAL-track-appmode.md, 2026-07-12 삭제)

구 'AI 추적 whitelist(track)' 검토안은 이 설계가 방향을 대체(자동 수확 + 사람 확정)
하여 문서 삭제. 아직 살아있는 세부 아이디어만 승계:
- **약속(commitment) 추적** — 주체(who)·기한(due) 있는 "내가/상대가 하기로 한 것"
  원장. 결정 원장과 대칭 구조 — 필요해지면 Phase 2+ 에서 decisions 패턴 재사용.
- **사용자 메모의 AI 컨텍스트 주입** — 프로젝트 도시에에 사람이 쓴 note 를 두고
  주간 증류 프롬프트에 함께 주입 (ai-rules.md 의 프로젝트판).
- **새 스레드 자동 제안** — "이 새 스레드, X 프로젝트에 붙일까요?" (Phase 2 배정의
  키워드·인물 매칭 보강안).
