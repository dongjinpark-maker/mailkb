# Mid-thread join — 합류 이전 대화의 보존

상태: **구현 완료** (2026-07-13)

## 1. 문제

스레드 중간에 수신인으로 추가되면(mid-join) 그 이전 메일은 두 겹으로 사라진다.

1. **물리 한계** — 내가 받기 전 메일은 내 사서함에 배달된 적이 없다. 어떤 수집기도 가져올 수 없다.
2. **인용 절단** — 유일한 사본은 "합류 시점에 받은 첫 메일"의 인용 체인인데,
   인용 제거(`extract_new_content` / `sanitize_html`)가 이를 삭제했다.

인용 제거의 철학은 "DB에 이미 있는 중복 제거"다. mid-join 첫 메일의 인용은
DB에 **없는** 내용이므로 중복이 아니라 유일본 — 예외여야 한다.

## 2. 판정: "스레드의 내 첫 보유분"

'대화의 진짜 첫 메일'은 관측 불가능하지만, 필요한 것은 **내 사서함 기준 첫
보유분**이고 이는 정확히 판정된다: 시간순 입력 하에서
`store._assign_thread` 가 **새 스레드를 생성하는 순간의 메일**이 곧 그것이다.

보존 트리거 = **새 스레드 생성 + 인용 체인 존재** (인용이 없으면 보존할 것도 없음).

### 전제: fetch 의 전역 시간순 (구현 포함)

구현 초기의 폴더 순차 순회(Inbox 전체 → Sent 전체)는 폴더 안에서만
시간순이라, 내가 시작한 스레드의 백필에서 상대 답장(Inbox)이 내 원 메일(Sent)
보다 먼저 들어와 오탐(내 메일이 뒤에 실물로 들어오는데 인용본을 보존)을
만들었다. `OutlookComSource.fetch` 가 두 폴더 스트림을 `heapq.merge` 로
병합해 전역 시간순을 보장한다 — `store.ingest` 의 "시간순 입력 가정"도 이로써
실제로 성립한다.

남는 오탐 하나(수용): 30일 밖 재개 스레드는 제목 매칭이 끊겨 새 스레드가
되고 인용이 보존된다 — 표시로는 맥락 복원이라 오히려 유익.

## 3. 두 층의 보존 (결정: 캡 없음)

| 층 | 형식 | 수명 |
|---|---|---|
| 텍스트(`new_content`) | `PRESERVED_MARK` 한 줄 아래 원문 그대로 | **영구** (캡 없음 — 프룬 대상 아님) |
| HTML(`message_html`) | `<details class='qfold'>` 접힘 (기본 닫힘) | 이미지 프룬과 동일 (`image_retain_days`) |

- 텍스트 층: 보존부의 `>` 줄·서명은 제거하지 않는다(체인 기록의 일부고, 서명
  절단 패턴이 체인 전체를 지울 수 있음). FTS·AI(요약/수확)·검색은 전문을 본다
  — 유일본이므로 AI 재료로 쓰는 것이 보존의 목적이다.
- **신호는 신규 작성분만**: 기한/요청/결정 정규식과 §1 '증발한 요청' 판정은
  `strip_preserved()` 로 마커 위쪽만 스캔 — 보존 인용 속 과거 요청은 이미
  소화된 대화라 개입 근거가 아니다.
- HTML 층: `_Sanitizer(preserve_quotes=True)` 가 절단 지점에서 버리는 대신
  열린 태그를 닫아 균형을 맞추고 폴드를 연다. 폴드는 메일당 하나(중첩 라벨은
  내용으로 흐름). 인용부 cid 이미지도 임베드된다(프룬이 회수).

## 4. 프룬과의 역할 분담

프룬은 표시용 HTML만 지우고 텍스트 층은 건드리지 않는다. 따라서:

- retain 기간 안: 원 서식 + 이미지 + 접힘(HTML 층)
- retain 경과 후: 웹 렌더러가 `PRESERVED_MARK` 를 **같은 접힘(qfold)으로
  재현** — 저장 증가 없이 렌더링만 늘어난다. 캡이 없으므로 프룬 후에도
  체인 전문이 텍스트로 남는다.

## 5. 기존 데이터 소급

소급 마이그레이션 없음 — 보존은 수집 시점 동작이라 새 메일부터 적용된다.
기존 스레드에 오는 새 답장은 스레드 매칭이 되므로 동작 변화 없음.
과거분까지 적용하려면 DB 삭제 후 `sync --full` (릴리즈 정책과 동일 절차).

## 6. 검토했으나 채택하지 않은 것

- **B안: 인용 체인을 파싱해 가상 메시지로 승격** — 인용 포맷 파편화(클라이언트
  ×언어×중첩)로 파싱 신뢰도가 낮고, 실패가 조용히 데이터를 오염시킨다.
- **C안: 타인 사서함 접근** — 권한·프라이버시로 불가.
- **첫 메일 HTML 프룬 예외(더 오래 보존)** — 프룬의 존재 이유가 이미지 용량
  회수인데 첫 메일이 가장 무거운 축. 텍스트 층 영구 보존으로 충분.
